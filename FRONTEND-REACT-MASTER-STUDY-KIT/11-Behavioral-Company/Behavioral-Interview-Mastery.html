P1 — Frequently Asked Questions

1
Tell me about yourself — structure your answer for a Senior Frontend role.P1
Basic Answer

Structure: Present (current role, key achievements) → Past (relevant experience, growth) → Future (why this role, what you want to learn). Keep it 2 minutes. For senior React roles: highlight specific React achievements, performance improvements, architecture decisions, and team leadership. Example: "I'm a Senior Frontend Engineer at X where I lead the migration from a legacy class-based React app to Next.js with Server Components, resulting in 40% performance improvement."

2
Tell me about a time you improved React app performance.P1
Basic Answer

STAR: Situation (app was slow with poor Core Web Vitals), Task (improve LCP from 4.2s to under 2s), Action (implemented Next.js with ISR, code splitting, lazy loading images, optimized bundle with code analysis, added virtualization for long lists), Result (LCP 1.8s, 35% higher conversion rate). Quantify everything.

3
Describe a conflict with a designer/PM about a technical decision.P1
Basic Answer

STAR: Situation (design wanted complex animation that caused performance issues on mobile), Task (find compromise), Action (proposed simplified animation using CSS transforms + GPU compositing, showed performance data on low-end devices, offered alternative that maintained visual quality), Result (designer agreed, animation ran at 60fps on all devices). Shows data-driven decision making.

4
Tell me about a production bug you fixed. Walk through the debugging process.P1
Basic Answer

STAR: Situation (users reporting data loss in form after upgrade), Task (find and fix root cause), Action (reproduced locally, used React DevTools to identify stale closure in useEffect, traced to missing dependency — used lint rule to catch, added comprehensive test), Result (bug fixed, added ESLint react-hooks/exhaustive-deps rule, mentored team on closure patterns). Shows systematic debugging + prevention.

5
Describe a time you mentored a junior developer.P1
Basic Answer

STAR: Situation (junior was struggling with React hooks and state management), Task (level up their skills), Action (paired programming sessions, code review focused on patterns, created internal guide on React hooks best practices, progressive complexity), Result (junior became productive, shipping features independently within 3 months, later mentored others). Shows leadership.

6
Tell me about a time you had to choose between different technical approaches.P1
Basic Answer

STAR: Situation (needed state management for large project), Task (choose between Redux, Zustand, Context), Action (evaluated based on team size, complexity, performance needs, built prototype with each, presented trade-offs with recommendations), Result (chose Zustand for simplicity + React Query for server state, team was productive within a week). Shows technical judgment.

7
How do you approach code reviews? What do you look for in React code?P1
Basic Answer

React-specific review checklist: Correct hooks usage (deps arrays, rules of hooks), proper key usage in lists, unnecessary re-renders, component composition (too big? extract), state management (is it server vs client state?), error handling, accessibility (semantic HTML, ARIA), prop types/TypeScript, test coverage for new logic. Give constructive feedback. Focus on correctness first, style second.

8
Tell me about a project you architected from scratch.P1
Basic Answer

STAR: Situation (greenfield project), Task (architect frontend), Action (chose Next.js + TypeScript + Tailwind, set up monorepo with shared packages, implemented feature-based folder structure, set up CI/CD with GitHub Actions, established coding standards, created component library), Result (shipped in 3 months, team of 6 was productive, 99.9% uptime). Shows architectural thinking.

9
Describe a time you failed on a project. What did you learn?P1
Basic Answer

STAR: Situation (chose wrong state management for scale), Task (fix before it blocked features), Action (acknowledged mistake, proposed migration plan, executed gradually with feature flags), Result (migration took 2 weeks but unblocked the team). Lesson: prototype more thoroughly before committing to architectural decisions. Self-awareness is key.

10
How do you stay current with the React ecosystem?P1
Basic Answer

Show genuine engagement: follow React core team (Dan Abramov, Sophie Alpert), read React RFCs, contribute to open source, experiment with new features (Server Components, React 19), read blogs (overreacted.io, react.dev), attend conferences (React Conf, React Summit), build side projects with new tech. Be specific about what you're currently exploring.

11
Amazon-specific: Give an example of Customer Obsession.P1
Basic Answer

STAR: Focus on customer impact measured by metrics. Example: "I noticed from analytics that users were abandoning checkout at 60%. I interviewed 5 users, identified the issue (form validation errors weren't clear), redesigned the form with inline validation and clear error messages, reducing abandonment to 25%." Connect to customer first.

12
Google-specific: How would you design YouTube's frontend architecture?P1
Basic Answer

Architecture: modular micro-frontends (Watch, Search, Recommendations, Studio). Server-side rendering for initial load, streaming for comments. Caching strategy for videos. Performance: virtual scrolling for infinite feed, custom video player with adaptive bitrate. State management: Zustand for UI state, React Query for server state. Offline support with Service Workers for watch history.

P2 — Commonly Asked Questions

13
How do you handle competing priorities with multiple stakeholders?P2
Basic Answer

STAR: Situation (PM wanted new features, EM wanted tech debt reduction, long list of requests), Task (balance priorities), Action (created RICE framework for prioritization, visual roadmap showing trade-offs, regular stakeholder syncs, transparent backlog), Result (team focused on high-impact work, stakeholders understood decisions). Shows stakeholder management.

14
How do you introduce a new technology to your team?P2
Basic Answer

Step by step: (1) Research and prototype — prove it works for your use case. (2) RFC document — propose with trade-offs. (3) Team discussion — gather feedback, address concerns. (4) Pilot — use on a low-risk project first. (5) Gradual adoption — create migration guide, pair with team members. (6) Retrospective — evaluate results. Shows technical leadership and emotional intelligence.

15
Tell me about a time you had to ship under a tight deadline.P2
Basic Answer

STAR: Situation (critical client demo in 2 weeks, normal estimate was 4), Task (deliver MVP), Action (identified core features, cut nice-to-haves, worked with PM to prioritize, focused on vertical slice first, deferred non-critical tech debt), Result (delivered on time, demo was successful, added remaining features post-launch).

16
Describe your ideal development workflow and tools.P2
Basic Answer

Tools: VSCode + Vim bindings, GitHub Copilot, Git with conventional commits, feature branch workflow, PR reviews with CI checks, linear/notion for project management. Testing: RTL + MSW + Playwright. CI/CD: GitHub Actions with lint → type-check → test → build → deploy. Deployment: Vercel with preview deployments. Monitoring: Sentry + DataDog RUM.

17
How do you handle technical debt?P2
Basic Answer

Follow the "scout rule" — leave code cleaner than you found it. Dedicated tech debt sprints (10-20% of time). Quantify debt in terms of developer velocity. Prioritize based on: what's blocking features, what's causing bugs, what's most painful. Track debt in the backlog. Communicate trade-offs to PMs. Some debt is okay — strategic debt vs irresponsible debt.

P3 — Occasionally Asked (Valuable Context)

18
How would you test Google Search's autocomplete feature?P3
Basic Answer

Test categories: debouncing (correct timing), API calls (correct params, error handling), rendering (suggestions appear, keyboard navigation works, click selects), performance (no unnecessary re-renders, memory usage), accessibility (screen reader announcements, keyboard-only navigation), edge cases (empty response, network failure, rapid typing), mobile (touch interactions, viewport).

19
How would you build a quality-first culture in your team?P3
Basic Answer

Actions: establish coding standards (documented), automated quality checks in CI (lint, type-check, test coverage), make testing easy (good test infrastructure), blameless post-mortems, knowledge sharing sessions, pair programming for complex features, celebrate quality wins. Lead by example — write tests, review thoughtfully, prioritize quality.

20
Microsoft-specific: How would you design Teams' frontend?P3
Basic Answer

Architecture: micro-frontends for different modules (Chat, Channels, Meetings, Files). Electron for desktop, React/TypeScript for web. Key challenges: real-time messaging (WebSocket), performance with large channel history, offline support (service worker + indexedDB), accessibility (screen reader compatibility), theming (enterprise branding), cross-platform consistency.
