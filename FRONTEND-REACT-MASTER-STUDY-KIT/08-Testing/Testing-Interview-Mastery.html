P1 — Frequently Asked Questions

1
What is the testing trophy vs testing pyramid? What should you test in React?IntermediateP1
Basic Answer

Testing pyramid: unit (many) → integration (some) → E2E (few). Testing trophy (Kent C. Dodds): integration tests are most important. React strategy: integration tests for component behavior (RTL), unit tests for pure functions/utils, E2E for critical user flows (Playwright). Focus on user behavior, not implementation details. Don't test internal state — test what the user sees and does.

2
What is React Testing Library? How does it differ from Enzyme?IntermediateP1
Basic Answer

RTL tests components from the user's perspective — queries elements like a user would (by role, text, label). Enzyme tested implementation details (state, props, instance methods). RTL is the officially recommended testing library. Principles: find elements by accessibility, test behavior not implementation, avoid testing internal state. Queries priority: getByRole > getByLabelText > getByPlaceholderText > getByText > getByTestId.

3
Explain the difference between unit, integration, and E2E testing.BeginnerP1
Basic Answer

Unit: test a single function/component in isolation (Jest + RTL). Integration: test how components work together, data flows (RTL + MSW). E2E: test complete user journeys in a real browser (Playwright, Cypress). Each level catches different bugs. For React apps: focus on integration tests (testing trophy), use E2E for critical paths, unit tests for complex logic only.

4
How do you test async operations in React components?IntermediateP1
Basic Answer

Use waitFor or findBy* queries (which retry until found). Mock API calls with MSW or jest.fn(). act() wraps state updates. jest.useFakeTimers() for timer-based async. For data fetching: mock at the network layer (MSW), not the fetch call itself. Test loading states, success states, and error states.

test('loads and displays users', async () => { render(); expect(screen.getByText('Loading...')).toBeInTheDocument(); await screen.findByRole('listitem'); expect(screen.getAllByRole('listitem')).toHaveLength(3); });
5
What is MSW (Mock Service Worker)? How does it improve testing?AdvancedP1
Basic Answer

MSW intercepts network requests at the Service Worker level (browser) or server level (Node.js for tests). Benefits: tests use real fetch calls (not mocked modules), works for integration and E2E tests, handles all HTTP methods, realistic request/response lifecycle. No need to mock fetch or axios — MSW handles everything.

6
How do you test custom hooks?AdvancedP1
Basic Answer

renderHook from @testing-library/react (v14+). Test initial state, state updates, and side effects. For hooks with context: wrap with provider using wrapper option. For hooks with side effects: assert that callbacks are called, effects run. Integration test (through components) is often better than testing hooks in isolation.

7
What is Playwright and how does it compare to Cypress for E2E testing?IntermediateP1
Basic Answer

Playwright (Microsoft): multi-browser (Chromium, Firefox, WebKit), auto-waiting, network interception, trace viewer, mobile emulation, API testing, built-in reporters. Cypress: limited to Chromium, not truly multi-tab, different architecture (runs in browser). Playwright is more popular for new projects (2024+). Both are great — Playwright is more powerful, Cypress has better DX for beginners.

8
What is mocking? When should you mock in React tests?IntermediateP1
Basic Answer

Mocking replaces real implementations with controlled fakes. Mock: network requests (MSW preferred), browser APIs (localStorage, fetch), third-party modules, timers (jest.useFakeTimers). DON'T mock: internal component dependencies, DOM queries, user interactions. RTL encourages minimal mocking — test real behavior. MSW mocks at the network boundary, which is the cleanest approach.

9
How do you test React context providers?AdvancedP1
Basic Answer

Create a wrapper component with the provider, render components inside it. Test that consumers receive correct context values, that updating context updates consumers, and error cases (consuming context without provider). Create a test helper: renderWithContext(ui, { providerProps }).

10
What are the different query methods in RTL? Priority order?IntermediateP1
Basic Answer

Priority: getByRole (accessibility) > getByLabelText (forms) > getByPlaceholderText > getByText (non-interactive) > getByDisplayValue (form values) > getByAltText (images) > getByTitle > getByTestId (last resort). Variants: getBy* (sync, throws if not found), queryBy* (sync, null if not found), findBy* (async, retries).

11
How do you test form submissions?IntermediateP1
Basic Answer

userEvent.type(input, 'value') (prefer userEvent over fireEvent — more realistic). userEvent.click(button). Assert API was called, success message appears, form resets. Test validation errors: submit empty form, check error messages. userEvent simulates actual browser events more accurately than fireEvent.

12
What is code coverage? What's a good target for React apps?BeginnerP1
Basic Answer

Coverage measures which code is executed during tests. Metrics: lines, branches, functions, statements. Good target: 80%+ for business logic, but don't chase percentages — focus on testing behavior. 100% coverage doesn't mean bug-free. Use --coverage flag. Coverage reports help identify untested code paths.

13
How do you test accessibility in React components?AdvancedP1
Basic Answer

jest-axe for unit tests: expect(await axe(container)).toHaveNoViolations(). Playwright's @axe-core/playwright for E2E. RTL encourages accessible queries (getByRole). Test: keyboard navigation, screen reader announcements, focus management, color contrast. Accessibility testing should be automated AND manual.

14
What is snapshot testing? When should you use it?IntermediateP1
Basic Answer

Snapshot testing captures rendered output and compares with stored snapshots. Good for: detecting unintended changes in output, component library consistency, large rendered outputs. Bad for: large/noisy snapshots, frequently changing components. toMatchSnapshot(). Update with -u flag. Use sparingly — prefer explicit assertions over snapshots for most cases.

15
How do you test Redux/Zustand connected components?AdvancedP1
Basic Answer

Wrap with provider in test. Redux: render(). Zustand: create fresh store for each test. Prefer integration tests: render the connected component, test user interactions, assert state changes. Test the complete flow: dispatch → state change → UI update. Mock API calls with MSW.

P2 — Commonly Asked Questions

16
What is Vitest? How does it compare to Jest?P2
Basic Answer

Vitest is the Vite-native test runner. Faster than Jest (no transpilation needed), compatible with Vite config, uses ESM. API-compatible with Jest (expect, describe, it, mock). Better DX — hot module replacement for tests. Growing popularity for new Vite + React projects. Migration from Jest is usually straightforward.

17
What is the act() function in RTL?P2
Basic Answer

act() ensures all state updates and effects are processed before assertions. RTL wraps most operations in act() automatically. You might need manual act() when: directly triggering state updates outside RTL, testing effects with timers, or integrating with non-React code. Warning: "not wrapped in act" means some state update happened outside React's batching.

18
How do you test routed components (React Router/Next.js)?P2
Basic Answer

React Router: render(). Next.js: next-router-mock for mocking useRouter. Test: navigation on click, reading params, handling query strings. For Next.js App Router: useRouter mock from next/navigation.

19
How do you handle visual regression testing in React?P2
Basic Answer

Tools: Percy, Chromatic (Storybook integration), Playwright Visual Comparisons (expect(page).toHaveScreenshot()). Chromatic is most popular for React (Storybook + visual regression). Set baseline screenshots, compare against them on CI. Handle: dynamic content (use fixed data), viewport differences, font rendering inconsistencies.

20
What are test fixtures and factories?P2
Basic Answer

Fixtures: pre-defined test data (hardcoded objects). Factories: generate test data dynamically (@faker-js/faker, @jackfranklin/test-data-bot). Factories reduce test brittleness — they generate unique data each test. Example: const user = buildUser({ role: 'admin' }) with sensible defaults. Factories make tests more maintainable and catch hardcoded-value assumptions.

21
How do you test error boundaries?P2
Basic Answer

Render a component that throws inside the error boundary. Spy on console.error to suppress expected error logs. Assert the fallback UI is rendered. jest.spyOn(console, 'error').mockImplementation(() => {}). Test: different error types, nested error boundaries, recovery from errors.

22
What is the jest.mock vs jest.spyOn difference?P2
Basic Answer

jest.mock(module) auto-mocks an entire module at the top of the file. jest.spyOn(object, method) wraps an existing method — can call through (spyOn(obj, 'fn').mockImplementation(...)). Use jest.mock for external modules, jest.spyOn for partial mocks of existing objects. Restore with mockRestore() (spyOn) or jest.restoreAllMocks().

23
How do you set up Playwright for React E2E testing?P2
Basic Answer

npm init playwright@latest. Config: playwright.config.ts with projects for Chrome/Firefox/WebKit, webServer (start Next.js dev), testDir. Use page.goto(), page.getByRole(), page.locator(). Assertions: expect(page.locator(...)).toBeVisible(). Test auth flows, navigation, and critical user journeys. Trace viewer for debugging failures.

24
What is TDD? Is it recommended for React development?P2
Basic Answer

TDD: write tests BEFORE implementation. Red-Green-Refactor cycle. Good for: pure functions, complex logic, bug fixes (write test that reproduces bug first). Less practical for: UI components (design often changes), exploratory development. Hybrid approach: TDD for business logic, test-after for components, E2E for critical flows.

25
How do you test Next.js Server Components?P2
Basic Answer

Server Components can be tested with RTL + Jest/Vitest. They render to React elements synchronously (no hydration needed). Test: data fetching (mock fetch), conditional rendering, and layout structure. For Server Actions: test the action function directly (it's a server-side function). Integration tests with Playwright are more realistic for full server/client interaction.

26
What is property-based testing? When would you use it?P2
Basic Answer

Property-based testing asserts properties that should ALWAYS hold true, then generates random inputs. Libraries: fast-check, jsverify. Example: "sorting a sorted array should return the same array." Finds edge cases you didn't think of. Best for: pure functions, data transformations, validation logic. Overkill for most UI tests.

27
How do you structure your tests for maintainability?P2
Basic Answer

Arrange-Act-Assert pattern. Group tests by feature/component. Use describe blocks for organization. Extract common setup (beforeEach). Use test data factories. Write test descriptions as sentences. Follow the "one assertion per test" for unit tests, multiple related assertions for integration. Keep tests close to the code they test (__tests__ or .test.tsx colocation).

28
How do you test component that uses useRef?P2
Basic Answer

Test behavior, not refs. If a component focuses input on mount: test that input is focused. If it measures DOM: test the effects of those measurements (layout changes). DON'T test the ref value directly — test observable behavior. Use jest.spyOn for imperative ref methods if needed.

P3 — Occasionally Asked (Valuable Context)

29
How do you test WebSocket or real-time features?P3
Basic Answer

Mock the WebSocket client. Test: connection established, messages received trigger UI updates, disconnection handled, reconnection works. Use jest.fn() for message handlers. For E2E: use a test WebSocket server (like ws library). Integration tests with MSW support WebSocket interception (experimental).

30
What are the costs of over-testing?P3
Basic Answer

Slower test suites (longer CI), brittle tests (break on refactoring), maintenance burden (updating snapshots, fixing mocks), false confidence (tests that pass but don't catch bugs), and opportunity cost (time spent testing instead of building). Balance: test behavior, not implementation. Integration tests > unit tests > snapshot tests. Delete flaky tests.

31
How do you implement test-driven bug fixes?P3
Basic Answer

1. Write a test that reproduces the bug (fails). 2. Understand the root cause. 3. Fix the code. 4. Test passes — bug is fixed and will never return. This prevents regression. The test becomes documentation for the edge case. Best practice for any confirmed bug.

32
What is Storybook and how is it used in testing?P3
Basic Answer

Storybook is a UI component explorer for developing and documenting components in isolation. For testing: @storybook/test for interaction testing, Chromatic for visual regression, Storybook play functions for component interactions. Storybook + Testing Library integration. Stories serve as both documentation and test cases.

33
How do you handle flaky tests? Strategies?P3
Basic Answer

Flaky tests pass sometimes, fail others. Causes: timing issues, test pollution, network dependency, async race conditions, order dependency. Fixes: use RTL's findBy* (retries), avoid setTimeout in tests, clean up between tests, mock network completely, isolate tests. Playwright auto-retries failed tests. Quarantine persistently flaky tests.

34
How do you test API integration tests without a real backend?P3
Basic Answer

MSW (recommended) — intercepts at the network level. json-server for quick mock APIs. miragejs for server mocking. Playwright can route network requests (page.route()). Each approach has trade-offs. MSW is the most realistic for integration tests — it uses the real fetch API and can be used identically in unit and E2E tests.

35
What is the role of CI/CD in testing?P3
Basic Answer

CI runs tests on every push/PR. Key practices: lint + type-check first (fast), unit/integration tests next (fast), E2E tests last (slow). Run E2E in parallel with sharding. Match CI environment to production (headless browser, same Node version). Fail CI on failed tests. Generate and publish test reports. Code coverage thresholds.
